RTOS LIFECYCLE:
Normaly MCU run on MSP (Main Stack) but when we want real time support every task need their special stack
so we MUST use PSP. It will not damage to MCU runcycle. All critical parts for MCU still run on MSP so if any task broke only this task efect MCU will not go hardfault etc

For using PSP We need to set up SPSEL bit from CONTROL Register. When MCU/board reset default SP_Main starting.
More clear:
Handler mode  all the time MSP
- Reset_Handler
- SysTick_Handler
- PendSV_Handler
- HardFault_Handler
- diğer IRQ'lar

our tasks PSP :)

PSP isolation does not prevent HardFault completely, but gives the system a recovery chance. If a task corrupts its own stack,
the HardFault handler still runs on MSP safely. Inside the handler we can read IPSR to identify which task caused the fault, then
mark that task as TASK_PAUSED and trigger a context switch to continue running other tasks without full system reset.

This is exactly why RTOS tasks must use PSP, not MSP. MSP is the last line of defense.

On writes, sets the PendSV exception as pending. On reads, indicates the

LDR:

Load from memory to register.

LDR Rd, [Rn, #offset]
      ▲    ▲    ▲
      │    ▲    └── how many bytes forward
           └─────── where is base register
      └──────────── result

Exp from my code:

ldr r1, [r0, #0] # read task->sp
r0 = task_t* (task_t* from c part on my code..)
[r0, #0] = task_t's offset 0 = sp field
r1 = task->sp

ldr r2, [r1, #24] read PC from stack
here :
r1 = starting point of exception frame
#24 = 6 × 4 byte forward
[r1, #24] = PC valuıe
r2 = task's start adress

C also they are same:
ldr r1, [r0, #0]  means  r1 = *((uint32_t*)r0);
ldr r2, [r1, #24] means  r2 = *((uint32_t*)(r1 + 24));
